# W1.5-04: load_llm_config silent failure on malformed config

## Intent
load_llm_config() has two bare except Exception blocks that silently
return LlmConfig() defaults on any failure — including malformed YAML
and pydantic validation errors. A user who misconfigures llm.max_tokens
gets default values with no indication anything is wrong. Worse: the
validator hot path (_default_validator, loop.py:730) calls this on every
validate pass — a config error there silently becomes a warn verdict,
which halts the task under unanimous policy with no actionable message.

Distinguish between:
- Missing config file → silent default is acceptable (file is optional)
- Malformed YAML → raise
- Pydantic validation failure (wrong types, out-of-range) → raise

## What to change
1. Add ConfigLoadError(Exception) to infrastructure/config.py (not
   core/interfaces.py — keep it local to infrastructure).

2. In load_llm_config():
   - Missing file (FileNotFoundError) → return LlmConfig() silently,
     same as today. Config file is optional.
   - YAML parse error (yaml.YAMLError) → raise ConfigLoadError with
     the file path and the yaml error message.
   - Pydantic validation error (ValidationError) → raise ConfigLoadError
     with the field name and the validation error message.
   - Other unexpected exceptions → raise ConfigLoadError wrapping the
     original exception.

3. Call sites:
   - build_protocol_graph (loop.py:1194) — let ConfigLoadError propagate.
     The existing try/except in run_cmd.py will surface it as
     "Error: Failed to build graph: <msg>" to the user. That's correct.
   - _default_validator (loop.py:730) — catch ConfigLoadError specifically
     and return ValidatorResult(severity="blocker", justification=
     "Config error: <msg>"). Not warn — blocker. A misconfigured validator
     should halt loudly, not escalate silently.
   - config_cmd.py — let it propagate, CLI will show the traceback. Fine
     for an interactive command.

## Acceptance criteria
- Missing config file still returns defaults silently
- Malformed YAML raises ConfigLoadError with actionable message
- Pydantic validation failure raises ConfigLoadError with field + message
- _default_validator returns blocker (not warn) on ConfigLoadError
- build_protocol_graph propagates ConfigLoadError to CLI error handler

## Testing
- Unit test: missing file → LlmConfig() defaults returned
- Unit test: malformed YAML → ConfigLoadError raised, message contains
  file path
- Unit test: invalid field value (e.g. max_tokens: "bad") →
  ConfigLoadError raised, message contains field name
- Unit test: _default_validator with mocked ConfigLoadError →
  ValidatorResult(severity="blocker") returned
- Existing config + engine test suite passes clean

## Constraints
- Touch only infrastructure/config.py and engine/loop.py (_default_validator
  call site only — do not refactor loop.py beyond this)
- Do not add ConfigLoadError to core/interfaces.py
- Do not fix the hot-path re-read performance issue in this ticket
